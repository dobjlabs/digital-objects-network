# Plonky2

[Plonky2](https://github.com/0xPolygonZero/plonky2) is the proof system at the bottom of the stack. zk-craft consumes it through pod2's `backend_plonky2` feature.

This page assumes basic familiarity with the proof-system landscape. A few terms used below in plain language:

- **Goldilocks field** — the small finite field Plonky2 does its arithmetic in (`p ≈ 2⁶⁴`). Small enough that arithmetic fits in a 64-bit register, which is why proving and verification are fast on consumer CPUs.
- **Poseidon hash** — an algebraic hash function defined directly in the field, so it's cheap to compute _inside_ a circuit. zk-craft uses it for object commitments, nullifiers, Merkle nodes, and predicate hashes.
- **FRI** — the commitment scheme Plonky2 uses to bind a proof to its committed data. It's what gives Plonky2 STARK-flavored characteristics (transparent setup, no trusted ceremony) while still producing succinct verifiable proofs.
- **Two-adicity** — a property of the field (Goldilocks has two-adicity 32) that determines the maximum size of FFT-friendly subgroups, which is what bounds circuit size before performance falls off a cliff.
- **Recursion** — proving inside a proof. A circuit can verify another Plonky2 proof; this is how MainPods compose, and how the [shrunk MainPod](/zk-stack/mainpod) wrapper works.

## Why Plonky2

Three properties matter for zk-craft:

| Property                                | Why it matters here                                                                                                                                                         |
| --------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Fast proving**                        | Action proofs are generated client-side on a consumer laptop. Single-digit seconds for simple actions.                                                                      |
| **Cheap recursion**                     | The MainPod is itself a composition of smaller proofs. The shrunk-MainPod wrapper is yet another recursive proof. Plonky2's recursion-friendly design makes both practical. |
| **Constant proof size after shrinking** | Once wrapped through the shrink circuit and compressed, the proof fits in an EIP-4844 blob regardless of how many statements the MainPod proved.                            |

There are no on-chain verifiers. The synchronizer is the verifier — every blob's proof is verified against canonical state during ingest.

## Field choice

zk-craft inherits Plonky2's defaults via pod2:

```rust
// from pod2 (backend_plonky2 feature)
pub use plonky2::field::goldilocks_field::GoldilocksField as F;
pub const D: usize = 2;
pub type FE = QuadraticExtension<F>;
pub type Hasher = PoseidonHash;
```

| Parameter            | Value                       | Notes                                                                                                     |
| -------------------- | --------------------------- | --------------------------------------------------------------------------------------------------------- |
| Native field `F`     | **Goldilocks**              | Prime `p = 2^64 - 2^32 + 1`. Fits in a `u64`, very fast Mont mul, two-adicity 32.                         |
| Extension degree `D` | `2`                         | Quadratic extension `FE = F[x]/(x² - W)` for FRI / SNARK soundness.                                       |
| Hash                 | **Poseidon**                | Algebraic hash, cheap inside the circuit. Used everywhere — object commitments, nullifiers, Merkle nodes. |
| Commitment scheme    | FRI (Fast Reed-Solomon IOP) | What makes Plonky2 a STARK-flavored SNARK.                                                                |

A `Hash` in pod2 is **4 field elements** (`HASH_SIZE = 4`). A `RawValue` is also 4 field elements. Object IDs, nullifiers, and Merkle roots are all `Hash` values in this representation.

## Proofs and proof artifacts

A Plonky2 proof carries three pieces of state:

| Artifact                         | Purpose                                                                                                                |
| -------------------------------- | ---------------------------------------------------------------------------------------------------------------------- |
| `ProofWithPublicInputs<F, C, D>` | The proof itself, plus its committed public inputs. Big — a typical MainPod proof is on the order of hundreds of KB.   |
| `CommonCircuitData<F, D>`        | The "shape" of the circuit: gate set, FRI parameters, public input layout. Same for every proof from the same circuit. |
| `VerifierOnlyCircuitData<C, D>`  | The minimum a verifier needs in addition to the common data: the verifier's public commitments.                        |

For verification, a verifier needs all three. The synchronizer caches the shrunk-MainPod's `CommonCircuitData` and `VerifierCircuitData` in the OS user-cache directory (`~/Library/Caches/pod2/` on macOS, `~/.cache/pod2/` on Linux) on first verification, so subsequent verifications are fast.

## Compressed proofs

Plonky2 has two on-the-wire representations for proofs:

- `Proof` / `ProofWithPublicInputs` — the natural form, used during recursion.
- `CompressedProof` — re-encoded to omit redundant FRI query openings (the verifier can reconstruct them from challenges). Much smaller; what zk-craft puts in the blob.

The blob payload uses `CompressedProof` for the `Plonky2` proof variant:

```rust
// common/src/payload.rs
pub enum PayloadProof {
    Plonky2(Box<CompressedProof<F, C, D>>),
}
```

## Where Plonky2 shows up directly

Most zk-craft code interacts with Plonky2 _through_ pod2. Direct touches are:

- **`common::shrink`** builds a Plonky2 wrapper circuit (`ShrunkMainPodSetup` / `ShrunkMainPodBuild`) that recursively verifies a MainPod proof and re-exposes the same public inputs.
- **`common::proof`** verifies the shrunk proof during blob ingest — `ProofParser::verify` does `decompress` → `verify` against the cached `verifier_circuit_data`.

Everything else — predicates, statements, the proof of `TxFinalized`, the persistent containers — lives in pod2.
